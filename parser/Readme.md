# LITHP Parser

Parser for the LITHP language, implemented using `ocamllex` + `ocamlyacc`.

## Files

| File | Role |
|------|------|
| `ast.ml` | Defines the `atom` and `exp` types (the AST). Also contains `show_atom` and `show_exp` for pretty-printing. |
| `lexer.mll` | `ocamllex` specification — same rules as the tokeniser, but opens `Parser` instead of `Main`. Auto-generates `lexer.ml`. |
| `parser.mly` | `ocamlyacc` grammar — token declarations, grammar rules, and AST construction actions. Auto-generates `parser.ml` + `parser.mli`. |
| `main.ml` | Entry point. Reads from stdin, runs the parser, prints the AST (one expression per line). |
| `test.ml` | Self-contained test suite. Parses strings directly and compares output to expected AST strings. |
| `makefile` | Build rules for all targets. |

**Generated files (do not edit):** `lexer.ml`, `parser.ml`, `parser.mli`

---

## How to Build and Run

```bash
# Build everything (parser binary + test binary)
make

# Run just the tests
make test

# Run the parser interactively (type LITHP, press Ctrl-D to finish)
./lithp_parser

# Parse a file
./lithp_parser < myprogram.lithp

# Clean all generated and compiled files
make clean
```

## Design Decisions

**`CADR_COMBO → Id`**
`car`, `cdr`, `cadr`, `caadr`, etc. are matched by the regex `c[ad]+r` in the lexer and stored as `Id` values, since they are identifiers (function names), not operators.

**`NIL` handled in `exp`, not `atom`**
`()` is the empty list, which has type `Ast.exp` (`L []`), not `Ast.atom`. Placing it in the `exp` rule avoids a type mismatch.

**`QUOTE` desugared to `(quote ...)`**
`'foo` is syntactic sugar for `(quote foo)`. The parser rule `QUOTE exp → L [A(Sym "quote"); exp]` handles the desugar at parse time, so the AST is always in canonical form.

**Keywords (`lambda`, `cons`, etc.) appear in `atom`**
These keywords can appear as the head of a list: `(lambda (x) x)`, `(cons a b)`. Since they sit in list position just like any other symbol, they need an `atom` rule that produces `Sym "..."`.


**`open Parser` in `lexer.mll`**
The lexer opens `Parser` (not `Main`) so it uses the token type generated by `ocamlyacc` from `parser.mly`. This is what connects the two generated files.
