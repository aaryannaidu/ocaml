# LITHP Type Checker

This directory contains a semantic analyzer (type checker) for the LITHP language. It receives an Abstract Syntax Tree (AST) generated by the parser and verifies its type correctness, safely handling LITHP's weak typing and operator overloading contexts without running the code.

## File Structure & Roles

1. **`types.ml`**: Defines the foundational algebraic data types. It introduces `TInt`, `TBool`, and `TList of int`. It also defines `TFunction(args_count, return_type)` for evaluating lambdas, and adds a highly flexible `TAny` placeholder for unresolved generic types.
2. **`ast.ml`**: Reused from the parser module. Serves as the structural blueprint that the type checker recursively evaluates.
3. **`env.ml`**: Defines the "typing environment" (symbol table). It acts as a dynamic dictionary `(string * typ) list` used to continuously map and track local variable identifiers to their respective types.
4. **`typechecker.ml`**: The core semantic validation engine. Built recursively using OCaml pattern matching. It receives an AST node and an environment, enforcing strong mathematical type rules and raising `TypeError` exceptions if arguments are illegal.
5. **`lexer.mll` & `parser.mly`**: Directly imported from the parser implementation. Generates tokens and assembles the base AST structure for the type checker.
6. **`main.ml`**: The program entry point. Funnels stdin strings into Lexer → Parser → Typechecker, and outputs the final inferred Types (or neatly catches any bubbled `TypeError`s to terminal output).
7. **`Makefile`**: Unifies the compilation pipeline, automating transformations from `ocamlyacc` and `ocamllex` before compiling the cohesive `typechecker` binary.

## Key Design Decisions

1. **Returning Lists of Types:** To natively handle LITHP's weakly typed overloading (e.g., the expression `()` acting as both `Bool` and `List(0)`), the `infer` evaluation function consistently returns a `typ list` bounding all valid interpretations. If parameters crash yielding an empty list, a type error is thrown.
2. **`TAny` Wildcards**: Function parameters are un-typed in their raw `(lambda (x) ...)` definition. Therefore, they are bound as `TAny` in the environment. `TAny` universally bypasses `match_types` constraints so that dynamic values passed into higher-order functions or list operators down the tree do not falsely fail standard validations.
3. **Strict Common `cond` Types**: A strict design decision was made to enforce consistency across all branches inside a `cond` block. Every branch is recursively resolved, and an intersecting logical filter ensures that *all* consequence branches resolve to an identical common type to prevent runtime casing ambiguity.
4. **Non-evaluating Quotes**: Evaluated explicitly so that `(quote a)` does not recursively type-check its inner contents (bypassing unbound variable errors entirely). It safely halts resolution and returns `[TBool]` to reflect its atomic syntax.
